# 星陨纪元 UI设计规范专家评审报告

## 总体评价

本设计规范文档结构完整、细节详实，星空紫金配色与修仙题材契合度较高，ATB战斗界面信息密度适中，状态机设计逻辑清晰。整体方案在Godot 4.6.2平台可实现性良好，但存在部分交互细节需要优化，特别是面板层级管理和移动端移植预留方面需提前规划。

---

## 优点

**1. 视觉风格差异化鲜明**

星空紫金配色方案与“星陨纪元”主题高度契合，深蓝黑底搭配灵力紫与金色高光，营造出深邃神秘的修仙氛围。发光效果模拟灵力流动的设计语言贯穿始终，具有较强的视觉辨识度。配色规范中节点类型用色区分清晰（普通战斗#6B6B8D、精英战斗#FFD700、BOSS#FF6B35），玩家可快速识别节点类型。

**2. ATB时间轴设计直观易理解**

将ATB行动顺序置于顶部60px区域，即将行动者高亮显示的设计符合玩家认知习惯。“行动就绪”状态文字提示配合发光边框，提供清晰的行动反馈。敌人血条三档颜色变化（绿→橙→红）配合脉动动画，在高压战斗中对玩家形成有效的危险警示。

**3. 状态机设计完整且安全**

12个状态的枚举定义覆盖完整，状态转换图清晰展示了MAIN_MENU→CHARACTER_SELECT→HUB→MAP→BATTLE的主流程。面板锁定规则（“面板打开时禁止场景切换”）配合ESC例外设计，兼顾操作安全与紧急暂停需求。附录提供的GDScript代码示例可直接用于生产环境。

**4. 快捷键体系符合PC玩家习惯**

全局快捷键布局合理（M地图、E装备、I背包、Q任务），战斗快捷键采用数字键1-6映射技能与物品，符合Steam平台玩家的肌肉记忆。快捷键视觉提示设计降低了新玩家的学习成本。

**5. 组件规范可复用性高**

按钮三级分类（主按钮160×48px、次要按钮120×40px、图标按钮40×40px）、道具格64×64px等尺寸规范为团队协作提供统一基准。渐变色值、圆角、动画时长等参数化定义便于程序员直接调用。

## 问题与建议

### 🔴 严重问题（影响核心体验）

**1. ATB战斗界面缺少目标选择机制描述**

问题：技能按钮区仅显示5个技能按钮，但未说明如何选择攻击目标。在3-5个敌人的场景下，目标选择逻辑至关重要。

建议方案：在技能按钮下方增加“目标选择区域”，设计目标锁定机制：
```
┌─────────────────────────────────────┐
│ 点击技能 → 高亮敌人头像 → 点击确认 │
└─────────────────────────────────────┘
```
或采用“默认选中当前目标+Tab循环切换”机制，并在技能按钮悬停时显示“选择目标”提示。

**2. 装备对比系统缺少详细交互流程**

问题：装备面板中提到“vs当前: +10攻击”对比显示，但未说明对比的触发条件、显示时机，以及多装备槽位的切换逻辑。

建议方案：补充以下交互规范：
- 点击背包内装备时自动弹出对比面板
- 对比内容包含：属性差值（绿色+/红色-）、词缀差异列表
- 新装备属性低于当前装备时，用红色高亮警告
- 批量替换时提供“一键换装”快捷操作

**3. 状态机存在HUB_PANEL_OPEN和MAP_PANEL_OPEN的逻辑歧义**

问题：这两个状态实际上是子状态，但文档中将其与主状态并列。在实际开发中，面板状态应作为主状态的属性而非独立状态，否则状态组合会导致状态机膨胀（如BATTLE_PANEL_OPEN是否需要？）。

建议方案：重构状态机设计，采用分层架构：
```
主状态: HUB | MAP | BATTLE | PAUSED
面板标记: is_panel_open: bool
面板类型: current_panel: PanelType (装备/背包/商店...)
```
避免状态数指数增长。

**4. 背包排序功能缺少分类与筛选逻辑**

问题：背包面板提到“排序：价值▼”，但未说明排序维度（价值/等级/类型/稀有度）、筛选机制（类型标签页切换是否包含搜索功能）、以及超过背包容量后的处理方式。

建议方案：补充背包交互规范：
- 排序维度：价值降序、等级降序、类型分组、稀有度星级
- 快速筛选：类型标签页（武器/护甲/饰品/消耗品/材料）+ 搜索框
- 容量管理：满容量时提示并高亮出售按钮，锁定拾取操作

### 🟡 重要优化（提升体验）

**5. 地图节点线性布局过于简单，缺乏Roguelike分支感**

问题：5节点纯线性布局（节点1→节点2→节点3→节点4→节点5）缺乏Roguelike的核心魅力——选择与分支。与《杀戮尖塔》等同类游戏相比，路线规划深度不足。

建议方案：考虑增加分支机制：
- 节点2分叉为节点2a和节点2b（可选）
- 分支路径用虚线连接，标注“可选路线”
- 不同分支通向不同事件类型（如2a通往采集点，2b通往商店）

**6. 快捷键体系未考虑游戏手柄支持**

问题：仅定义了键盘快捷键，但Steam平台约30-40%的Roguelike玩家使用手柄。缺少手柄快捷键规范将限制部分玩家的体验。

建议方案：补充手柄映射方案：
```
LT/RT: 快速切换武器
A/X: 确认/使用
B/Circle: 返回/取消
LB/RB: 切换标签页
十字键: 移动选择焦点
```

**7. 词缀汇总面板信息量可能过载**

问题：词缀汇总面板同时展示元素加成、攻击词缀、防御词缀、特殊词缀四个分类，在装备词缀较多时（假设10+词缀），单屏信息密度过高，影响阅读效率。

建议方案：设计分组折叠机制：
- 默认展开当前装备词缀
- 其他来源词缀（buff/食物等）折叠显示
- 按属性类型（攻击/防御/功能）分页或使用子标签页切换

**8. 锻造系统缺少材料获取途径说明**

问题：锻造面板展示了“玄铁×3 + 玄铁×3 + 玄铁×3 = 陨铁×1”的合成公式，但未说明玄铁的获取途径（采集/怪物掉落/商店购买），玩家可能面临“知道配方但无法合成”的挫败感。

建议方案：在锻造面板增加“材料来源提示”：
- 悬停材料格时显示获取途径列表
- 标注各途径的获取概率或价格
- 不足材料用红色标记并显示缺口数量

### 🟢 建议改进（锦上添花）

**9. 音效规范建议增加战斗节奏提示**

问题：战斗音效列表完整，但未说明音效与ATB系统的配合（如ATB充能时的循环音效、即将行动时的警示音）。Roguelike战斗节奏紧张，音频节奏提示可大幅提升沉浸感。

建议方案：补充以下音效设计：
- ATB充能：循环粒子音效（0.5s间隔）
- ATB≥80%：加速充能音效
- 即将行动（第一位）：短促提示音
- 完美时机触发：特殊音效+画面特效

**10. 节点动画脉动效果建议增加强度分级**

问题：当前规范中“当前位置”统一使用“边框4px脉动发光”，但未考虑BOSS节点等高危情况的强化提示。

建议方案：增加脉动强度分级：
- 普通节点：边框2px脉动，周期1.5s
- 精英节点：边框3px脉动+金色粒子，周期1.2s
- BOSS节点：边框4px脉动+橙色警告光晕+心跳音效，周期1.0s

**11. 建议增加设置项详细清单**

问题：文档提到“暂停菜单→设置”包含“音频/画面/控制”子选项，但未详细说明可配置项。Steam玩家对设置项的丰富程度较为敏感。

建议方案：补充设置项清单：
- 音频：主音量、BGM音量、SFX音量、语音音量
- 画面：分辨率、帧率上限（全屏/窗口）、画质预设、屏幕滤镜强度
- 控制：鼠标灵敏度、震动强度、快捷键自定义

**12. 胜利/失败面板建议增加录像回放入口**

问题：Roguelike游戏战斗回放是重要的学习工具，但规范中未提及战报或录像功能。玩家失败后无法回顾战斗过程，失去了“从失败中学习”的核心体验。

建议方案：在DEFEAT面板增加：
- 战斗统计：回合数、伤害输出、治疗量、暴击次数
- 关键节点：战斗时间线标记
- 可选：“回放战斗”入口（存储最后一场战斗的完整数据）

## 开放式问题

**问题A：ATB系统是否需要加入“等待”选项？**

《勇气与传说》《最终幻想10》等ATB游戏提供“等待”指令，允许玩家在敌人行动前主动跳过当前回合以等待更好的时机。这一机制可以增加策略深度，但也可能拉长战斗时间。建议评估战斗节奏目标后决定。

**问题B：Roguelike游戏是否需要成就系统UI？**

文档提及“成就”按钮，但未设计成就面板详情。成就系统在小团队项目中可能优先级较低。建议考虑：成就面板是否可以简化为“进度追踪+通知弹窗”模式，而非独立面板？

**问题C：是否需要设计“新手引导”系统？**

当前规范面向有经验的开发者团队，假设玩家熟悉Roguelike和ATB机制。但首次接触此类游戏的玩家可能需要教程。建议评估：是否需要设计“可跳过的入门教程”UI流程？

**问题D：地图节点数量是否可动态调整？**

当前设计为固定5节点线性流程。建议考虑：不同游戏阶段（前期/中期/BOSS战）是否需要不同长度的节点链？这样可以提升重复可玩性。

**问题E：Steam Deck适配考量**

Steam Deck的掌机模式和桌面模式操作差异较大。当前快捷键设计基于桌面键盘，掌机模式需要额外的触控/陀螺仪操作方案。建议提前规划Steam Deck适配策略。
