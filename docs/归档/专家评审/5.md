## 二、代码质量与架构评审

### 2.1 架构设计（基于文件结构推断）

```
godot_project/
├── data/               # 数据层（.tres资源文件）
│   ├── characters/     # 角色定义
│   ├── realms/         # 境界定义
│   ├── equipment/      # 装备模板
│   ├── affixes/        # 词缀定义
│   ├── materials/      # 材料定义
│   └── permanent/      # 永久增幅定义
├── systems/            # 系统层（逻辑实现）
│   ├── attributes/     # 属性计算
│   ├── equipment/      # 装备生成
│   ├── affixes/        # 词缀效果+共鸣
│   ├── map/            # 地图生成
│   ├── collection/     # 采集逻辑
│   └── combat/         # 战斗管理
├── entities/           # 实体层
│   └── player/         # 玩家实体（属性/境界/装备）
├── scenes/             # 场景层（UI/视图）
│   ├── main.tscn       # 入口
│   ├── game.tscn       # 主游戏
│   ├── battle/         # 战斗场景
│   ├── map/            # 地图场景
│   ├── zone/           # 据点场景
│   └── ui/             # 各类面板
└── autoload/           # 全局单例
```

**优点**：
- **清晰的分层架构**：数据/系统/实体/场景分离，符合 Godot 最佳实践
- **数据驱动**：角色、境界、词缀、材料均使用 `.tres` 资源文件，便于调整和扩展
- **模块化**：系统之间通过信号/全局单例通信，耦合度低

### 2.2 潜在代码风险点（需开发团队自查）

| 风险点 | 说明 | 建议 |
|--------|------|------|
| **装备生成性能** | 每次掉落都动态生成装备实例（含随机技能槽、随机词缀），高频战斗下可能产生GC压力 | 使用对象池复用装备实例，或预生成缓存 |
| **词缀效果叠加顺序** | 多个词缀修改同一属性（如伤害加成）时，叠加顺序可能影响最终数值 | 明确加成类型：加法/乘法/最终乘算，统一处理顺序 |
| **共鸣系统刷新时机** | 装备更换时重新计算共鸣，需确保触发逻辑不会遗漏 | 在 `Player._on_equipment_changed` 中统一刷新 |
| **时砂系统与子弹时间冲突** | 时砂为真暂停（2秒），子弹时间为0.2x慢动作，同时触发时需明确优先级 | 时砂期间禁用子弹时间，或时砂覆盖子弹时间 |
| **存档数据版本兼容** | Phase 1 未包含存档系统，后续加入时需考虑数据结构迁移 | 使用 `ResourceSaver` 保存玩家数据时加入版本号字段 |

### 2.3 数据驱动程度评估

**优秀**：
- 角色、境界、词缀、材料均使用 `.tres` 资源，策划可独立配置
- 装备生成算法基于模板+随机规则，无需硬编码

**可改进**：
- 敌人数据（HP/攻击/技能）目前可能硬编码在战斗系统中，建议也改为 `.tres` 资源文件
- 掉落表（不同敌人掉落概率/物品池）可配置化，便于后续平衡调整

## 三、数值平衡评审

### 3.1 ATB速度与软上限250

**验证**：
- 50级基础敏捷 142.5 → ATB = 100 + 142.5×3 = 527.5
- 溢出 = 527.5 - 250 = 277.5 → 伤害加成 = 277.5%
- 加上装备词缀（疾风+10%）和增幅器（速度×1.3~2.0），溢出可超过400%

**评估**：
- 溢出伤害加成过高，可能导致敏捷属性碾压体质/精神
- 但玩家需要达到50级+特定装备才能实现，属于“后期爽点”

**建议**：
- 将溢出伤害加成改为 **溢出部分×0.5%**，降低线性增长
- 或设置溢出伤害加成上限为 200%

### 3.2 战斗时长目标

| 战斗类型 | 设计目标 | 实测预估值（基于文档公式） | 评估 |
|----------|----------|---------------------------|------|
| 普通怪 | 3-8秒 | 约5-7秒 | ✅ 合理 |
| 精英怪 | 10-15秒 | 约12-18秒 | ⚠️ 略偏高 |
| BOSS | 30-40秒 | 约35-50秒 | ⚠️ 可能略长 |

**建议**：
- 适当降低精英怪 HP 倍率（3x → 2.5x），或增加精英怪易伤机制
- BOSS 战中引入“阶段暴露弱点”机制，缩短有效战斗时间

### 3.3 装备掉落率

| 敌人类型 | 掉落率 | 每局预计掉落数（假设一局杀50怪+5精英+1BOSS） | 评估 |
|----------|--------|---------------------------------------------|------|
| 普通怪 | 20% | 约10件 | ✅ 充足 |
| 精英 | 50% | 约2-3件 | ✅ 合理 |
| BOSS | 100% | 1件 | ✅ 合理 |

**建议**：无问题，当前掉落率可提供足够的装备迭代。

### 3.4 词缀共鸣门槛

| 共鸣等级 | 条件 | 效果 | 评估 |
|----------|------|------|------|
| 基础 | 2个同系 | +5%伤害 | ✅ 容易触发，数值合理 |
| 进阶 | 3个同系 | +15%~20%伤害 | ✅ 需刻意搭配，奖励适中 |
| 高级 | 4个同系 | +30%~35%伤害 | ⚠️ 门槛高，奖励略保守 |

**建议**：
- 高级共鸣增加额外特效（如“物理极致：攻击无视10%防御”），而非纯数值
- 速度共鸣高级条件 ATB>300 在后期容易达到，可调整为 >350

| ATB软上限200→250 | 已修改 | ✅ 已实现 | 合理 |
| 时砂局内成长性 | 已修改（杀敌恢复） | ✅ 已实现 | 优秀设计 |
| 子弹时间触发条件 | 已修改（完美时机） | ✅ 已实现 | 合理 |

**关于用户确认“保持不变”的两项**：
1. **区域地图5等级**：57-70级BOSS对于前期玩家确实过高，但设计意图是“后期回来挑战”，不强制前期通关。建议在UI中明确提示“推荐等级”，避免玩家挫败。
2. **阵营永久绑定**：仍存在玩家“选错后悔”风险。建议在阵营选择界面增加详细预览和“确认弹窗”，或允许第一次免费更换。

## 五、Phase 2 开发建议

### 5.1 优先级排序

| 优先级 | Phase 2 内容 | 原因 |
|--------|-------------|------|
| **P0** | 敌人AI多样化 | 当前敌人行为单一，影响战斗趣味 |
| **P0** | 套装完整效果 | 装备驱动核心，应与词缀系统联动 |
| **P1** | 开放战斗舞台 | 多敌人同时战斗，提升ATB策略性 |
| **P1** | 势力敌人系统 | 增加地图随机性和阵营关系 |
| **P2** | 词缀联动UI提示 | 改善玩家Build构建体验 |

### 5.2 技术债务处理

- **存档系统**：Phase 2 必须实现，需支持玩家数据（角色/装备/永久增幅/传送门激活状态）的保存与加载
- **性能优化**：装备生成对象池、战斗特效对象池
- **本地化准备**：所有UI文本抽取为可翻译字符串

### 5.3 平衡性微调建议

- **敏捷属性**：当前贡献ATB速度+暴击率+闪避率，收益过高。考虑降低敏捷对ATB的加成（3→2.5）
- **能量系统**：能量恢复手段较少（仅击杀+0.5、暴击回能词缀），可增加基础每回合恢复
- **防御减伤公式**：`减伤=防御/(防御+100)` 在防御>200后收益递减明显，可考虑改为 `防御/(防御+150)` 降低前期防御收益

### 5.4 测试重点

| 测试项 | 方法 |
|--------|------|
| 速度溢出极限 | 使用控制台给角色+200敏捷，验证伤害加成是否封顶 |
| 词缀组合冲突 | 同时装备“玻璃大炮”和“铁壁·荆棘”，验证伤害加成与护盾反弹是否正常叠加 |
| 共鸣系统刷新 | 频繁更换装备，验证共鸣效果实时更新 |
| 突破容错 | 故意属性不达标，尝试试炼突破和星尘重塑 |
| 死亡惩罚 | 多次死亡，验证星尘扣除20%和装备保留逻辑 |

### 优点
- Phase 1 实现完整，与设计文档高度一致

> **Phase 1 已达到可发布内部测试版本的标准。建议团队先进行内部 playtest，收集数值反馈，再进入 Phase 2 开发。重点测试战斗节奏和装备Build多样性。**
