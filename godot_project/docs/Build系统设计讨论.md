### 1.3 核心问题：词缀效果是"死"的

当前实现中，词缀只是数据定义，**没有任何实际效果**：

```gdscript
# data_manager.gd - 词缀只是存在字典里
_create_affix(&"fire_damage_15", "炽焰", "火属性伤害+15%", Enums.AffixType.DAMAGE, 0.15)

# 但战斗计算时没有应用这个加成
# battle_manager.gd 的伤害公式：
var total_damage = int(base_damage * atb_bonus * (1.0 + kinetic))
# ❌ 没有 fire_damage_15 加成
```

## 二、需要讨论的关键问题

### 2.1 装备的随机性设计

**当前问题：**
- 装备生成时从 `possible_affix_ids` 池中随机选择
- 词缀池太小，随机结果缺乏多样性
- 没有装备品质变体（蓝装vs紫装只是颜色不同？）

**需要确定的：**

| 问题 | 选项 | 影响 |
|------|------|------|
| 随机时机 | 掉落时生成 / 局外预生成 / 两者都有 | 局外成长感 |
| 词缀数量 | 固定 vs 范围 vs 完全随机 | _build多样性 |
| 词缀数值 | 固定 vs 浮动范围 | 平衡性难度 |
| 装备等级 | 无等级 / 等级压制 / 随机装等 | 刷装动力 |
| 词缀重锻 | 无 / 锁定部分 / 完全重随 | 付费点/肝度 |

**我的建议：**
```
掉落时随机 → 每件装备1-4个词缀（稀有度决定）
词缀数值浮动 → 基础值±20%
可锁定重锻 → 消耗资源重随部分词缀
```

### 2.2 词缀效果系统

**当前问题：**
- 16个词缀定义了，但`apply_affix_effect()`系统不存在
- 词缀分6类，效果代码分散在各处

**需要设计的：**

| 词缀类型 | 效果应用时机 | 实现方式 |
|---------|-------------|---------|
| 伤害类 | 伤害计算时 | `final_damage *= (1 + affix_value)` |
| ATB类 | ATB变化时 | `speed_bonus += affix_value` |
| 能量类 | 能量操作时 | `max_energy += affix_value` |
| 元素反应类 | 反应触发时 | 检查条件，激活强化 |
| 特殊类 | 各种条件 | 触发器模式 |
| 防御/生存类 | 受伤计算时 | `damage_taken *= (1 - affix_value)` |

**建议：统一 `AffixSystem` 处理**
```gdscript
# systems/equipment/affix_system.gd
class_name AffixSystem

func apply_affix(affix_id, entity, context):
    match affix.type:
        DAMAGE: apply_damage_affix(...)
        ATB: apply_atb_affix(...)
        # ...

func calculate_damage_modifiers(base, skill, target):
    # 所有伤害相关词缀汇总
    var total_element_damage = 0
    var total_crit_damage = 0
    # ...
```

### 2.3 技能搭配性设计

**当前问题：**
- 技能来自装备，但只有3个技能/武器
- 链技（chain_skill）只是同技能id延续，不跨技能

**技能来源设计需要讨论：**

| 来源 | 当前 | 替代方案 |
|------|------|---------|
| 武器决定全部 | 3技能/武器 | 武器只给1-2核心技能 |
| 饰品给技能 | 没有实现 | 饰品给1个强力技能 |
| 护甲给技能 | 2技能/护甲 | 护甲只给防御相关 |
| 通用技能池 | 无 | 所有装备共享技能池 |

**技能联动设计：**

```gdscript
# 方案A：元素反应联动
skill fire_attack + skill ice_skill → 触发蒸发反应

# 方案B：ATB联动
skill A (ATB加速) + skill B (高伤害) → ATB加速后接高伤害

# 方案C：动能联动
动能满时 skill A 强化 → 动能技能树

# 方案D：套装联动
装备2件套 + skill → 获得额外效果
```

**我的建议：重点发展方案A（元素反应联动）作为核心Build路径**

### 2.4 灵石系统（完全缺失）

**设计文档中的灵石：**
- 6个灵石：火/冰/雷 各分上下两级
- 效果：元素伤害+、ATB速度+
- 槽位：3个灵石槽

**需要确定的：**

| 问题 | 选项 |
|------|------|
| 灵石获取 | 地图掉落 / 商店购买 / 合成 / 全部 |
| 灵石随机性 | 固定词缀 / 随机词缀 / 随机数值 |
| 灵石是否可升级 | 否 / 同元素合成升级 / 随机升级 |
| 灵石槽解锁 | 剧情解锁 / 护甲决定 / 任意槽位 |

### 2.5 套装系统

**当前设计：3套4件套装**

| 套装 | 路线 | 2件 | 3件 | 4件 |
|------|------|-----|-----|-----|
| 量子丹田 | 科技 | 能量恢复+30% | 释放技能ATB+15% | 溢出能量→伤害+50% |
| 雷劫 | 修真 | 雷伤+25% | 15%概率重置ATB | 完美时机伤害×3 |
| 磐石 | 中立 | 防御+20% | 15%概率免疫 | 护盾碎→ATB+30% |

**需要讨论的：**

| 问题 | 选项 |
|------|------|
| 套装数量 | 3套 / 5-6套 / 每章节不同 |
| 套装稀有度 | 全部紫装 / 混搭稀有度 / 随机 |
| 套装效果强度 | 强（必须）/ 中等（可选）/ 弱（锦上添花） |
| 非套装组合 | 有无"伪套装"奖励（同路线X件） |

## 三、Build系统核心问题

### 3.1 我们的Build深度够吗？

参考成功游戏：

| 游戏 | Build复杂度 | Build驱动力 |
|------|------------|------------|
| Slay the Spire | 卡牌组合 | 卡牌池+遗物+牌组构建 |
| Hades | 武器变体+恩盖 | 武器+祝福+契约 |
| 我们的方案 | 装备+词缀+套装 | ？？？ |

**当前设计的Build来源：**
```
武器(3技能) + 护甲(2技能) + 饰品(0-1技能) + 灵石(0) + 词缀(5-8个) + 套装(3件套效果)
```

**问题：技能来源太少，词缀效果太弱**

### 3.2 理想中的Build循环

```
局内Build:
  装备选择 → 词缀效果 → 技能搭配 → 元素反应 → 战斗验证
       ↑                                      ↓
       └──────── 掉落更好的装备 ───────────────┘

理想状态:
  - 每件装备都有独特价值（不是简单的属性高低）
  - 词缀组合产生1+1>2的效果
  - 元素反应是Build的核心桥梁
  - 套装效果是Build的终极目标
```

### 3.3 需要新增的Build内容（建议讨论）

| 内容 | 优先级 | 说明 |
|------|--------|------|
| 词缀效果系统 | P0 | 否则词缀是废数据 |
| 元素反应词缀 | P1 | 6个词缀强化反应 |
| 灵石系统 | P1 | 6个灵石+3槽位 |
| 饰品技能 | P1 | 8饰品应给技能 |
| 武器变体 | P2 | 每武器2-3变体 |
| 套装效果实现 | P1 | 当前只有文字 |

## 四、讨论议程

请确认以下问题我们需要重点讨论：

### 必选项（影响核心玩法）
1. **装备随机性深度**：掉落时随机 vs 预生成，装等/词缀数值范围
2. **词缀效果实现**：统一系统 vs 分散代码
3. **技能来源分配**：武器3技能是否太多？饰品给技能？

### 重要项（影响Build多样性）
4. **灵石系统**：是否实现？实现方式？
5. **元素反应作为Build核心**：是否围绕反应设计词缀和技能？
6. **套装数量和效果**：3套够吗？效果强度？

### 可选项（影响付费/肝度）
7. **词缀重锻系统**：是否允许玩家锁定重随？
8. **装备等级/压制**：是否加入？
9. **局外Build保留**：死亡后保留部分装备/词缀？
